这题主要是用Flask 后端的认证方式，所有需要登录的路由只从请求头里拿 X-User，**如果头不存在，默认值是字符串 0；若等于 'anonymous' 就重定向登录。前置 Go “Authorizer” 反向代理：统一把请求转发到 127.0.0.1:5000 的 Flask，并在转发前删除来路的 Authorization / X-User / Cookie，然后根据 JWT（或 Cookie 里的 token）自己加一个 X-User**：没登录就设为 anonymous，并起了一个本地 127.0.0.1:4444/sign 签名服务，Flask 登录成功后会去这里拿 token。

漏洞点一

当 X-User: 0时就是admin权限了，直接导致无头=admin(0)的逻辑。

但是现在的问题是用go做了反向代理，我们无法直接访问后端的Flask

漏洞点二

Go 的 httputil.ReverseProxy 会遵循 HTTP/1.1 规范：把请求头 Connection: 指定的所有 header 名称当作 hop-by-hop 头，在转发时剥掉。

也就是说我们发送Connection: X-User时，虽然反代的 Director 先给请求添加了 X-User: anonymous，但在真正发给后端前，ReverseProxy 会因为 Connection: X-User 而把这个 X-User 头删掉。后端 Flask 收到的请求因此**没有 X-User**，触发默认值 '0' → 直接把我们当成 uid=0 的 admin。

  • EvaluationService
    • 用 SpelExpressionParser + TemplateParserContext（模板语法 #{ ... }）。
    • 返回 "Result: " + String.valueOf(result)；异常则返回 "Error: ..."。
  • SpelConfig
    • 构造 SimpleEvaluationContext（受限版 SpEL，上来就把危险操作关了：不支持 T(), new, 反射、任意方法调用等）。
    • 只注册了一个 PropertyAccessor：SecurePropertyAccessor extends ReflectivePropertyAccessor，且 canRead(...) 永远返回 false —— 也就是禁止反射属性读取。
    • 另把变量 #systemProperties 设为 System.getProperties()（后面能直接取出来作为对象）。

就算 SpEL 被锁死了，**Controller 仍然会无条件尝试读取并回显 user.home 下的flag.txt**。这是典型的“业务逻辑层敏感信息泄露”（IDOR/不当访问控制的一种表现）。

也就是说虽然像 #{T(java.lang.Runtime).getRuntime().exec('id')} 这类危险 SpEL，在 SimpleEvaluationContext 下会直接报错（类型引用/方法调用被禁），但是在 check 接口里把读取并回显服务器本地敏感文件放在求值之后无条件执行，造成信息泄露：只要文件存在、就回显第一行，无需任何权限判断或白名单校验。

思路关键点：

  • /fetch 的 meta refresh 解析是直接在响应正文里用正则扫 "<meta http-equiv='refresh' content='0;url=...'>"；它并不要求响应 Content-Type: text/html。
  • 我们可以让“内层” /fetch 故意报错（比如把 url 传成根本不是 URL 的一段 meta 标签），这样它会返回 jsonify({"error": "Upstream request failed: ...<meta ...>..."})。
    JSON 字符串里 单引号不需要转义，因此那段 meta 标签会原样出现在响应文本里，被“外层” /fetch 的 META_REFRESH_RE 捕获并当作下一跳。
  • 如此套娃 6 层 http://127.0.0.1:5000/fetch?url=<meta...>，第 7 跳指向 http://127.0.0.1:5000/_internal/secret，最后一步由于 Content-Type: application/json 但正文不是 JSON，解析报错 + 已走 ≥6 步 ⇒ /fetch 把最后一个响应原文回显，SECRET 即可到手
  • 有了 SECRET，即可打SSTIRCE
