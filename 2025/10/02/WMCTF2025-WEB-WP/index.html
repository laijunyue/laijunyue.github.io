源码

主要漏洞点就是eval(payload, {'__builtin__':{}})处的代码执行

但是需要str(key1) == str(key2)，key2是伪随机生成的

参考https://blog.csdn.net/m0_57291352/article/details/119655082

通过RandCrack中有一个方法是predict_getrandbits()，在给出的随机数数量多时，可以预测下一个随机数

而通过访问/api随便给出key的值即可得到key2的随机数

之后利用 python 继承链来绕过no builtins

后来发现是将"__builtin__": {}而不是"__builtins__": {}，内置模块在 Python 2 中它叫 __builtin__，在 Python 3 中更名为 __builtins__，而题目是python3，相当于这啥也没过滤，该不会是出题人写错了吧。。。

EXP

pdf2text

给了提示

Web “pdf2text” added hint: Search “pickle.loads” in pdfminer package and try to reach it

那么肯定是从pickle.loads入手打pickle反序列化

这里我是真的蠢，DockerFile的requirements.txt里给了pdf三方库是pdfminer.six，我下成了pdfminer导致怎么都找不到pickle.loads

Ctrl + Alt + F7 一步一步寻找调用

在CMapDB 类下的_load_data函数有调用pickle.loads(gzfile.read()))

重点看参数gzfile.read()也就是gzip.open(path).read()，filename需要是 .pickle.gz 结尾，它在两个目录下查找并读取 .pickle.gz文件：环境变量 CMAP_PATH（默认 /usr/share/pdfminer/）与包内 pdfminer/cmap/

name 是可控的，来自 PDF 中的 CMap 名，或 ToUnicode CMap 流里的 usecmap 操作数

因此可以利用name打路径穿越将path = os.path.join(directory, filename)改为/app.uploads/(name)

而name又是由pdf文件中的/Encoding决定，此外pdf中使用 # 来转义一个16进制数, 这样就可以使用 ..#2F构造出 ../ , 避免与pdf中符号 / 表示名字对象冲突

EXP

建一个恶意 .pickle.gz 文件

新建一个1.pdf，将 /Encoding 字段的值改为

/Encoding /..#2F..#2F..#2F..#2F..#2F..#2F..#2F..#2F..#2F..#2F..#2Fapp#2Fuploads#2Fshell

先传恶意 .pickle.gz ，再传pdf触发pickle反序列化即可getshell

Ezparquet

Rustdesk笑传之change client backend

不知道如何连接靶机？你应该在你的电脑上安装 RustDesk 客户端，或者使用它的第三方 Web 客户端实现（需要http部署，而非https，否则无法访问ws端口），按照 readme.md 文件中的说明设置 ID/中继 (relay) 服务器，并连接到靶机网页中给出的 RustDesk ID。

如果你不想设置本地的调试用 RustDesk ID/中继服务器，我可以给你 RustDesk 服务器 47.96.76.236的密钥（Key）：bAdW+pgLg9iPFgbOXldxwi5fUWTZHXbwamBw3AAFm5w=。但是，知道这个Key实际上可能会妨碍你的解题思路，在解决挑战时，不知道这个密钥可能会更好。
（仅拥有Key不能直接连接到靶机，因为没有人会点击同意会话请求。同时，连接到靶机也不能进行任何操作，因为所有远程控制功能被禁用了。想想看为什么redis服务器存在。）

阅读RustDesk客户端和服务端源码，尝试连接至RustDesk 对端客户端，经过RustDesk ID服务器和中继服务器，但是绕过Key的使用。在这个过程中你可能会发现一些有趣的事情。
如果你需要一个开始点，我建议查找Key错误时的”Key mismatch”字符串。客户端什么情况下会输出这个字符串？
靶机的RustDesk客户端版本是1.4.1，服务端版本是1.1.14

关于绕过Key验证的更多信息：Key只在RustDesk ID服务器的handle_punch_hole_request函数中检验，客户端中不检验。但是，可以不使用这个函数创建中继连接，即从始至终不发送PunchHoleRequest请求，直接发送RequestRelay请求。
RustDesk 中继服务器中有检查Key的类似代码，但是默认情况下不检查。

本题目的目标是，利用某种限制很大的SSRF（服务端请求伪造）并且是RustDesk的漏洞，访问127.0.0.1的Redis，利用Redis主从同步写入so执行RCE（远程代码执行）。你需要一个rogue redis server 。
