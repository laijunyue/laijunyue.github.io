分析入口点

拿到题目，有附件最好。有些题目就算不给附件源码也并非是黑盒，可以通过路径穿越等方式把源码拉取下来，例如[网鼎杯 2020青龙组]FileJava。

题目比较常见的是Spring Boot框架。找到入口点，分析反序列化的具体方式。CTF中的Java反序列化一般是三种方式：

​ 1. 原生反序列化

​ 2. Jackson/FastJson反序列化

​ 3. 框架自写的反序列化（Hessian2/Kryo）

原生反序列化入口点分析

以**[MTCTF2022]easyjava**为例：

使用了自写的MyObjectInputStream类来处理反序列化，一般来说都会重写resolveClass方法对黑名单类进行一次判断：

这种黑名单判断是无法绕过的，其原理在于原生反序列化流程会创建一个desc（类描述符），其name属性是类的全限定名，resolveClass方法对desc里的name属性进行黑名单检测，这里的desc是我们无法自行修改的。

还有一种黑名单对String进行包名检测：

这个可以用TF-8 Overlong Encoding绕过

那对于resolveClass方法对desc.name进行黑名单检测，一般的思路是如何的呢？

  1. 最好是0day，但是常规来说也没人专门捏着0day往这里砸吧；

  2. 其次是黑名单里没写的类，比如UncaistRef或者其外面的代理类（这个类的优点我下面会讲）；

  3. 再者是打二次反序列化，不走题目给的resolveClass逻辑······但是往往一些比较常用的二次反序列化类（Signedobject、RmiConnector）也放不出来，自己多积累几个。

  4. 最后就是看看题目给的一些类。

Jackson/FastJson反序列化

部分CTF题目有用到这种反序列化，比如Bugku-CTF-Java Fastjson Unserialize、**[VNCTF2021]realezjvav等。但是原理比较简单，无非起一个远程服务，然后反序列化com.sun.rowset.JdbcRowSetImpl**等类触发一个RMI或者LDAP请求。这里就不做赘叙。

框架自写的反序列化（Hessian2/Kryo）

无论是Hessian2还是Kryo，其原理都是在反序列化过程中，如果反序列化对象是Map类，都会调用对应的put方法。

UnicastRef类

新学到了一个Java JRMP 反序化，RMI 依赖的通信协议为 JRMP(Java Remote Message Protocol ，Java 远程消息交换协议)，该协议为 Java 定制，基于 TCP/IP 之上，RMI 协议之下，当需要进行 RMI 远程方法调用通信的时候要求服务端与客户端都为 Java 编写。

这个协议就像 HTTP 协议一样，规定了客户端和服务端通信要满足的规范，RMI 底层默认使用的 JRMP 进行传递数据，并且 JRMP 协议只能作用于 RMI 协议。

利用UnicastRef类，它有以下几个优点：

  1. JDK自带的，对依赖没有要求

  2. 该类有readExternal方法，相当于readObject方法，无需为寻找调用链烦恼

  3. jrmp服务端响应的数据走系统的反序列化流程而不是题目重写的MyObjectInpuct()

  4. 黑名单禁止该类？我有更多的方法（下文讲）。

缺点：

  1. 本质上是打一个JRMP请求，不能断网，环境必须出网（现在大部分比赛几乎都是不出网）
  2. 黑名单常客，绕过：https://blog.hackpax.top/JavaDeserializeLabs/#0x07-Lab7

shiro反序列化

shiro反序列化可以说是每年护网的热门了，其实shiro框架本身是没有安全问题的，但是shiro会产生安全问题的主要原因是因为其可以让用户反序列化任意类，如果在使用shiro框架的项目中有上述的CC调用链，那么攻击者就可以利用其反序列化恶意链进而RCE

shiro的主要作用就是进行身份验证

shiro反序列化的流程

首先用户登录，并勾选”Remember Me”，shiro会将用户信息进行序列化

序列化用户信息 –> AES加密 –> Base64编码 –> 得到rememberMe的cookie值

当下次用户访问时

获取rememberMe的cookie值 –> Base64解码 –> AES解密 –> 反序列化

所以payload 构造的顺序就是

恶意链–>序列化–>AES加密–>base64编码–>发送cookie

其实不难看出，漏洞利用的关键有两点

第一，也是最为关键的就是，如何得到AES加密的key

第二，服务器是否有可以被利用的调用链

shiro反序列化漏洞主要分为Shiro-550和Shiro-721

复现可参考https://www.cnblogs.com/h0cksr/p/16189780.html

Shiro-550

影响版本：Apache Shiro < 1.2.4

Shiro-550的AES加密密钥默认被硬编码在源码中，密钥为kPH+bIxk5D2deZiIxcaaaA==，且使用已知密钥碰撞，只要有足够密钥库（条件较低）就可暴力破解出密钥，也就是说Shiro-550的利用条件是比较简单的

Shiro-721

影响版本：Apache Shiro 1.2.5、1.2.6、1.3.0、1.3.1、1.3.2、1.4.0-RC2、1.4.0、1.4.1

相较于Shiro-550，Shiro-721的利用条件就苛刻多了，Shiro-721的AES加密的key基本猜不到，系统随机生成，但虽采用随机密钥加密，但AES-128-CBC模式易受Padding Oracle Attack（填充提示攻击）影响。攻击者可利用有效的RememberMe Cookie作为Padding Oracle Attack的前缀，构造恶意RememberMe Cookie值，实现反序列化漏洞攻击，也就是说其利用需要正常登录的RememberMe Cookie

小插曲

如果对方的WEB服务是用Spring boot搭建，那么如果其配置不当，就会出现Spring boot的Actuator信息泄露，其中可能就会有一个/actuator/heapdump堆内存转储文件，它可以提供有关应用程序运行时内存使用情况的详细信息，里面会存放当前运行的WEB程序的敏感信息，就比如ShiroKey

所以说信息收集很重要！！！
